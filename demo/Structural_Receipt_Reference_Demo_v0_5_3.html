A deterministic Structural Receipt result states what the declared structure supports under one frozen purpose profile. It does not authenticate the issuer, prove that a physical transaction occurred, or authorize payment, reimbursement, posting, tax treatment, return acceptance, or another downstream action.
Core relation: same canonical input + same frozen profile → same supported result
Structural Receipt v0.5.3 is a bounded local reference implementation with portable verifier-facing evidence bundles. Exported bundles are designed for separate local verification without requiring a shared account or network service. Purpose resolution carries no automatic execution authority. The reference does not execute payments, approve reimbursement, post accounting entries, certify tax treatment, authenticate issuers, prove physical delivery, or constitute legal or production assurance.

Adoption Paths
Structural Receipt can begin on one side of an exchange without requiring coordinated platform replacement.
Adopt on one side. Verify on the other. Integrate only when useful.
Producer-Side Adoption
Existing system → Structural Receipt sidecar → portable evidence bundle

The producing system can keep its current receipt or transaction record and add a Structural Receipt sidecar. A receiving party can reconstruct the exported bundle through a separate verifier without sharing the producer's application.

Verifier-Side Adoption
Existing structured source → declared bridge or adapter → WRAPPED / DERIVED Structural Receipt

A verifier can adopt the resolution model without requiring the source system to become a native Structural Receipt producer. Provenance and semantic-equivalence limits remain explicit.

Native End-to-End Adoption
Native producer → portable bundle → Structural Receipt-aware verifier

When both sides adopt the structure directly, the same identities, purpose profiles, evaluation context, lineage, and authority boundaries can move across the exchange.
