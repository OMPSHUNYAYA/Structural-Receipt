1 Purpose

Structural Receipt is governed by declared purpose-specific resolution.

The current architecture represents each purpose profile as identified canonical structure rather than leaving the governing contract only inside implementation code.

The governing relation is:

`the rules governing the question are themselves declared identified structure`

Executable resolver logic still exists.

The manifest defines the bounded contract that the implementation claims to execute.

4 Purpose Manifest Set Identity

The current architecture also computes a deterministic identity over the current identified purpose-profile set.

The exact current set subject is:

`purpose_manifest_set_subject = {"profile":"SR-PURPOSE-MANIFEST-1-D01","purposes":[{"purpose":...,"purpose_profile_id":...},...]}`

The `purposes` array is sorted deterministically by `purpose`.

Each array entry contains exactly:

`purpose`

`purpose_profile_id`

The set identity is:

`purpose_manifest_set_id = SHA256(canonical(purpose_manifest_set_subject))`

The set identity allows an evidence bundle or verifier to distinguish:

`same receipt under same complete purpose-profile set`

from:

`same receipt under a changed purpose-profile set`

The current profile for this behavior remains:

`SR-PURPOSE-MANIFEST-1-D01`

6 Current Conflict Handling

The current profile manifests identify required dependencies.

The current resolver separately determines whether available evidence or lineage creates a conflict.

The architecture preserves:

`missing required dependency -> INCOMPLETE`

`incompatible required evidence -> CONFLICT`

`resolved negative outcome != unresolved state`

A future selective-disclosure profile is expected to add explicit result-affecting and conflict-relevant disclosure closures.

That future disclosure contract is not part of the current frozen profile.

11 Current Context Requirements

The following current purposes require:

`as_of`

`WARRANTY_EVIDENCE`

`RETURN_ELIGIBILITY_EVIDENCE`

`LINEAGE_CURRENT_STATE`

If the required `as_of` context is absent, the current resolver does not silently substitute the ambient current clock.

The governing relation is:

`time-relative question + explicit as_of -> deterministic supported result`

not:

`time-relative question + ambient now -> hidden result dependency`

The hostile-input corpus includes a frozen missing-`as_of` case.


12 Ambient Time Prohibition

A deterministic purpose profile must not silently use the current wall clock as an undeclared source of resolution authority.

This allows a result to remain reproducible later.

The determinism relation is:

`same receipt subject + same purpose profile identity + same declared evaluation context -> same supported result`

Time-relative evaluation is therefore compatible with deterministic reconstruction when the relevant time is explicit input.

The current browser editor likewise avoids using the ambient wall clock to generate new item identifiers. Editor-created identifiers use a monotonic local `ITEM-n` sequence. This keeps newly created receipt structure free from hidden wall-clock input while remaining separate from the declared `as_of` evaluation context.

20 Selective Disclosure Integration Direction

A future Purpose Disclosure Contract is expected to reference the purpose profile identity.

The future relation is:

`purpose profile identity -> disclosure contract identity -> complete result-affecting disclosure closure`

The architecture should preserve:

`minimum sufficient disclosure != convenient positive subset`

and:

`WITHHELD != MISSING`

The current profile-manifest specification does not yet define a frozen selective-disclosure rule language or commitment scheme.


21 Conformance Direction

- deterministic manifest identity;

- missing required `as_of`;
- changed currency profile;

- execution authority change;

22 Public Explanation

A purpose profile is something a reader can inspect, identify, compare, and independently reconstruct.

The strongest relation is:

`the rules governing the question are themselves declared identified structure`

The strongest determinism relation is:

`same receipt subject + same identified purpose profile + same declared evaluation context -> same supported result`

The strongest trust boundary is:

`a purpose profile determines a bounded answer; it does not create authority beyond its declared scope`


23 Portable Purpose-Profile Artifacts

The v0.5.3 release publishes each current purpose manifest as a standalone JSON artifact under:

`profiles/`

The folder contains one artifact for each current purpose plus one profile-set artifact.

The standalone files preserve the same identified manifest subjects already used by the browser and verifier implementations.

Therefore:

`embedded manifest subject == published standalone manifest subject`

within the current release.

The release does not treat a filename as profile authority.

The authority remains the identified canonical manifest subject and its reconstructed `purpose_profile_id`.

The complete current set reconstructs to:

`purpose_manifest_set_id = 20c55cd83bce5eb9590c757fc972880c11126249a2ed033f1f3b49249de6e381`

This enables implementations to inspect and exchange the governing profile data without reverse-engineering the browser source.


24 v0.5.3 Interchange Boundary

A portable evidence bundle binds the current purpose manifests, evaluation context, trust context, structure identities, lineage identities, and purpose receipts.

A verifier may therefore check:

`same receipt structure + same profile identities + same declared context -> same supported reconstructed result`

A bridge-generated WRAPPED Structural Receipt does not silently gain native-origin authority.

The current origin rule remains:

`NATIVE != WRAPPED != DERIVED`

and:

`successful adapter transformation != complete semantic equivalence`

The current bridge explicitly preserves:

`semantic_equivalence = NOT_ASSUMED`

The purpose-profile architecture remains usable across native and adapted sources because origin and provenance are part of the declared structure rather than hidden implementation state.
