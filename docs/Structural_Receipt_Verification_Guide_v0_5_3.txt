1 Purpose

This guide defines the recommended verification procedure for the Structural Receipt v0.5.3 repository.

It covers:

- browser release audit;
- frozen corpus reconstruction;
- standalone local bundle verification;
- separate Python reconstruction;
- bridge self-test;
- corpus identity checks;
- exact file-byte SHA-256 checks.

The current verification architecture demonstrates bounded reconstruction within the declared v0.5.3 profiles.

It does not establish independent third-party certification, issuer authenticity, real-world occurrence, legal validity, tax correctness, accounting correctness, payment settlement, or operational execution authority.

11 Corpus Identity Versus File Identity

Two different identity questions must remain separate.

`corpus_id`

identifies the canonical semantic corpus subject under the declared profile.

`file SHA-256`

identifies the exact published file bytes.

A repository release should preserve both.

A changed file with a deliberately regenerated internal corpus ID is still a different published file.

Therefore exact release verification should pin file hashes separately from semantic corpus IDs.


12 SHA256SUMS

The repository root contains:

`SHA256SUMS.txt`

It is the authoritative exact-file checkpoint for the selected executable artifacts in the current release:

`demo/Structural_Receipt_Reference_Demo_v0_5_3.html`

`verify/Structural_Receipt_Standalone_Verifier_v0_5_3.html`

`verify/Structural_Receipt_Independent_Verifier_v0_5_3.py`

`bridge/Structural_Receipt_Bridge_v0_5_3.py`

Other identity domains use their own declared identities rather than duplicating a second file-hash registry:

`semantic corpora -> corpus_id`

`purpose rules -> purpose_profile_id + purpose_manifest_set_id`

`evidence bundles -> bundle_id`

`verification evidence -> reconstructed identities + recorded result`

On Windows, an individual file may be checked with:

`certutil -hashfile filename SHA256`

Compare the resulting digest with the applicable line in `SHA256SUMS.txt`.

The checksum file should be regenerated whenever the release intentionally changes an artifact covered by that checkpoint.

Do not silently replace a frozen artifact while retaining an old checksum.

`same supported subject + same frozen profile set + same declared evaluation context -> same reconstructed identities and supported states across the demonstrated project-produced implementation paths`

The release does not claim:

- independent third-party verification;
- production certification;
- issuer authenticity;
- physical transaction occurrence;
- legal validity;
- tax correctness;
- accounting correctness;
- payment settlement finality;
- execution authority.

The final trust relation is:

`verification != authentication != approval != execution`
