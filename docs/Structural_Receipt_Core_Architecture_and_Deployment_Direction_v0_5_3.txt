1 Executive Summary

Structural Receipt is a deterministic architecture for turning a receipt from a passive transaction document into a portable, purpose-resolvable evidence structure.

The project begins from a simple pain point:

`one transaction should not have to be understood from scratch every time it is used`

A purchase may pass through merchant systems, payment systems, customer archives, expense workflows, accounting processes, return workflows, refund workflows, warranty processes, audit processes, and other downstream applications.

The same transaction may therefore be rendered repeatedly, extracted repeatedly, interpreted repeatedly, reconciled repeatedly, classified repeatedly, and reconstructed later from fragmented evidence.

Structural Receipt asks a narrower and more explicit question:

`what is this exact available structure sufficient to establish for this exact declared purpose?`

The current reference release evaluates nine bounded purposes under identified canonical manifests.

It separates:

`receipt data != universal transaction truth`

`issuer declaration != independent payment evidence`

`payment evidence != delivery evidence`

`delivery evidence != expense eligibility`

`expense eligibility != accounting classification`

`accounting classification != accounting posting`

`integrity != authenticity`

`authenticity != real-world occurrence`

`resolution != approval`

`approval != execution`

The current release also extends the adoption model beyond a closed environment.

A producer can adopt Structural Receipt while a recipient verifies a portable bundle locally.

A recipient can adopt the bridge and verifier path while the source system remains unchanged.

An existing JSON-producing system can be wrapped through a declared adapter manifest without claiming complete semantic equivalence.

The practical release relation is:

`existing system -> bridge or sidecar -> portable Structural Receipt evidence -> local verifier`

This means coordinated bilateral migration is not required before either side gains bounded utility.


2 Core Structural Model

The current conceptual object is:

`SR = (C, E, D, L, O, V)`

where:

`C = claims and canonical purchase declaration`

`E = available evidence`

`D = declared purpose dependencies`

`L = linked lineage events`

`O = origin and adapter provenance`

`V = versioned schemas, rules, profiles, contexts, and identity subjects`

The current purpose resolver receives:

`P = identified purpose profile`

`X = declared evaluation context`

and computes:

`R_P = resolve(SR, P, X)`

A purpose result may contain:

- evaluation status;
- resolution state;
- purpose outcome;
- authority state;
- execution authority;
- reasons;
- evidence references;
- required dependencies;
- satisfied dependencies;
- missing dependencies;
- conflicting dependencies;
- dependency coverage;
- purpose profile identity;
- evaluation context identity;
- purpose result identity;
- trust-context fields.

The architecture separates:

`resolution_state`

from:

`purpose_outcome`

from:

`authority_state`

from:

`execution_authority`

A resolved negative result is still resolved.

A resolved proposal is still only a proposal.


3 Primary Dependency-Elimination Model

The primary dependency is repeated downstream interpretation.

The common pattern is:

`transaction evidence -> application-specific interpretation -> local conclusion`

repeated across many systems.

Structural Receipt proposes:

`declared evidence structure + identified purpose profile + declared context -> deterministic supported state`

The dependency statement is:

`repeated application-specific interpretation no longer remains the sole authority over what transaction evidence supports`

Operational applications remain.

Workflow remains.

The authority relationship changes.

8 Origin and Provenance Classes

The current origin classes are:

`NATIVE`

`WRAPPED`

`DERIVED`

A native Structural Receipt is created from declared transaction structure at origin.

A wrapped Structural Receipt is produced through a declared adapter from an existing structured source.

A derived Structural Receipt is reconstructed from a presentation or manually supplied source.

The governing separation is:

`NATIVE != WRAPPED != DERIVED`

The current browser import boundary treats these as a closed origin-class set. Unsupported imported origin-class values are rejected before admission into the active receipt state, and displayed origin labels are escaped defensively.

A successful transformation does not silently become a claim of complete semantic equivalence.

The current bridge preserves:

`semantic_equivalence = NOT_ASSUMED`


9 Canonicalization and Structural Identity

The active canonicalization profile is:

`SR-CANON-1-D01`

The strict input profile is:

`SR-JSON-INPUT-1-D01`

The canonical pipeline is:

`validated supported subject -> canonical JSON -> UTF-8 -> SHA-256 where the identity profile requires hashing`

Object keys are deterministically ordered within the supported domain.

Array order is preserved unless a higher-level identity subject explicitly normalizes a semantic set before canonicalization.

The current profile performs no implicit Unicode normalization.

Duplicate object names and trailing non-whitespace JSON input are rejected by the strict import boundary.

Canonicalization does not decide which fields belong to an identity subject.

Therefore:

`identity subject selection != canonicalization`

11 Declared Evaluation Context

The current evaluation-context identity subject contains:

`profile = SR-EVALUATION-CONTEXT-1-D01`

`as_of`

`jurisdiction_profile`

`currency_profile`

Time-relative purposes must not silently depend on ambient current time.

The relation is:

`time-relative question + explicit as_of -> deterministic supported result`

not:

`time-relative question + hidden current clock -> undeclared result dependency`

The current time-relative purposes requiring `as_of` include warranty evidence, return eligibility evidence, and lineage current state.

The browser editor also avoids using ambient wall-clock values to mint new item identifiers. New editor-created item identifiers use a monotonic local `ITEM-n` sequence derived from the active item set and local sequence state.

This is an implementation hygiene rule distinct from purpose evaluation, but it keeps wall-clock dependence out of newly created receipt structure.

13 Non-Destructive Lineage

A supported return, refund, or credit creates a linked event rather than silently rewriting the original purchase.

The relation is:

`change -> linked lineage event`

The current architecture separates:

`receipt_id`

from:

`lineage_graph_id`

`current_lineage_state_id`

Duplicate lineage-event identities are rejected as a conflict boundary.

19 Open-Boundary Adoption Architecture

A major v0.5.3 objective is to reduce the need for coordinated adoption.

The release supports three adoption modes.

Producer-side sidecar adoption:

`existing transaction system -> Structural Receipt sidecar -> portable evidence bundle`

The receiver can verify locally without adopting the producer application.

Verifier-side adoption:

`existing structured source -> declared bridge -> WRAPPED Structural Receipt -> local purpose resolution or verification`

The source system does not need to change first.

Native bilateral adoption:

`native Structural Receipt producer -> portable bundle -> Structural Receipt-aware verifier`

The adoption principle is:

`one adopting side -> immediate bounded utility`

`two adopting sides -> richer interoperability`

`many adopting sides -> broader portability`


20 Sidecar Deployment Model

Structural Receipt does not require replacing the visible receipt or existing system of record.

A deployment may preserve:

`must we replace the current system?`

to:

`can we add a portable structural evidence layer at a boundary?`

This is the preferred incremental adoption model.

22 Adapter and Interchange Principles

Structural Receipt should not replace mature financial-document ecosystems unnecessarily.

The current interoperability principles are:

1. preserve source identifiers;
2. preserve source artifact identity;
3. classify origin explicitly;
4. preserve unmapped source fields;
5. declare mapping warnings;
6. do not silently promote derived or wrapped data to native authority;
7. keep transport separate from semantic resolution;
8. keep payment evidence separate from payment execution;
9. keep accounting proposal separate from accounting posting;
10. preserve explicit unsupported states when a source cannot be mapped safely.

The compact interchange contract is:

`portable bundle in -> deterministic reconstruction out`

A receiver does not need to adopt the producer database, user interface, or cloud service.

25 Security and Abuse Boundaries

Structural Receipt provides deterministic structural reconstruction within declared profiles.

It does not itself provide:

- secure key management;
- issuer authentication;
- anti-fraud guarantees;
- malware isolation;
- universal input-size safety;
- legal admissibility;
- production access control;
- confidentiality guarantees.

The current browser release includes permanent regressions for inert rendering of hostile user-controlled markup across the tested display paths. The import boundary also rejects unsupported origin-class values before active-state admission, and origin labels are escaped at the verification display boundary.

These controls demonstrate the tested reference paths; they are not a claim of universal browser sandboxing or malware isolation.

The current strict JSON boundary rejects duplicate object keys and trailing data.

The current pure deterministic hashing path processes public structural identity subjects.

Future secret-bearing commitment or authenticity mechanisms require explicitly defined cryptographic implementation boundaries rather than being inferred from the current public identity-hashing path.


26 Privacy and Retention Boundary

Structural Receipt can contain sensitive transaction information.

The architecture therefore preserves:

`structural identity != privacy guarantee`

`non-destructive lineage != mandatory indefinite plaintext retention`

`selective disclosure != anonymity`

`selective disclosure != unlinkability`

Deployment-specific obligations concerning storage, retention, deletion, access control, encryption, and jurisdiction-specific data rights remain outside the generic architecture claim unless separately profiled.


27 Purpose-Limited Disclosure Direction

Purpose-relative resolution naturally leads to purpose-relative disclosure.

The future question is:

`what is the minimum complete result-affecting evidence that must be disclosed for purpose P?`

The intended dependency-elimination statement is:

`full-document disclosure -> no longer the sole path to purpose-specific verification`

Future profiles may include:

`SR-COMMIT-1-D01`

`SR-DISCLOSE-1-D01`

`SR-PDC-1-D01`

The key law is:

`minimum sufficient disclosure = complete result-affecting dependency closure`

A verifier must not resolve from a convenient positive subset while withheld committed evidence could change the result.

Future disclosure semantics should preserve:

`WITHHELD != MISSING`

and separate:

`semantic receipt identity`

from:

`commitment root identity`

Those profiles are not implemented or frozen in v0.5.3.

A related future diagnostic direction is a Structural Impact Trace:

`mutation -> affected dependency closure -> affected identity closure -> affected purpose-receipt closure`

This would explain which supported identities and purpose receipts are affected by a change while preserving independent subjects that remain unchanged.

It is not part of the frozen v0.5.3 implementation.


28 Authenticity Evidence Extension

32 Claim Boundary

Structural Receipt does not by itself prove:

- that a physical purchase occurred;
- that goods were delivered;
- that a payment settled;
- that an issuer identity is legitimate;
- that a tax treatment is correct;
- that an accounting treatment is correct;
- that an expense must be reimbursed;
- that a warranty claim must be approved;
- that a refund is legally required;
- that a transaction is free from fraud;
- that a signed claim is true;
- that a receipt is acceptable in every jurisdiction;
- that a successful adapter transformation is semantically complete;
- that project-produced verification is third-party certification;
- that the current reference implementation is production ready.

The current bounded architecture claim is:

`declared receipt structure + declared evidence + declared lineage + identified purpose rules + declared context -> deterministic supported result within the declared profile`

The current bounded verification claim is:

`same supported subject + same frozen profiles + same declared context -> same demonstrated reconstruction across the current project-produced implementation paths`


33 Relationship to the Shunyaya Ecosystem

Structural Receipt follows the wider structural principle that operational workflows may remain while no longer retaining sole authority over resolution.

Its core dependency shift is:

`repeated downstream interpretation -> no longer the sole authority over what transaction evidence supports`

The project keeps the structural object, purpose resolver, authority boundary, and evidence reconstruction explicit.

Its role is not to eliminate every downstream system.

Its role is to establish a portable structural boundary through which downstream systems can ask bounded questions without repeatedly rebuilding transaction meaning from presentation alone.

18. `change -> linked lineage event`

19. `original receipt remains addressable through supported lineage`

20. `profile rules are identified declared structure`

21. `one-side adoption can produce bounded utility`

22. `verification can occur without adopting the producer application`

23. `full-document disclosure is not assumed to be the permanent minimum for every future purpose`


36 Recommended Public Positioning

Structural Receipt should be described as:

`a structural resolution layer between transaction evidence and downstream conclusions`

The strongest practical statement is:

`A receipt should not have to be understood from scratch every time it is used.`

The strongest adoption statement is:

`Adopt on one side. Verify on the other. Integrate only when useful.`

The strongest verification statement is:

`producer assertions are not verification authority; supported identities and states are reconstructed`

The strongest claim boundary is:

`do not claim more than the structurally reconstructed evidence can support`

Structured invoicing and signed-document mechanisms commonly emphasize origin, integrity, authenticity, or compliance. Structural Receipt addresses a separate question: what the available transaction evidence is sufficient to support for an identified purpose under declared profiles and context.

Application-centered transaction workflows commonly optimize capture, storage, classification, or processing inside an operational environment. Structural Receipt emphasizes portable reconstruction outside the producer application while preserving that downstream policy remains locally governed.


37 Final Architecture Summary

Structural Receipt v0.5.3 establishes a bounded architecture in which transaction evidence can be declared once, resolved differently by purpose, preserved through supported lineage, and reconstructed across project-produced verification paths.

It also establishes an adoption surface that does not require coordinated migration before either side can gain utility.

A source system may continue to exist.

A bridge may wrap existing structured data while preserving provenance and limitations.

A producer may export a portable evidence bundle.

A recipient may verify that bundle locally without the producer application.

Purpose profiles are published as standalone identified artifacts.

Examples make the cross-boundary paths reproducible.

The current architecture therefore combines:

`purpose-specific deterministic resolution`

`+ explicit incompleteness and conflict`

`+ identified governing profiles`

`+ declared evaluation context`

`+ explicit trust and authority boundaries`

`+ origin-aware evidence`

`+ non-destructive lineage`

`+ portable evidence bundles`

`+ separate reconstruction paths`

`+ bridge-based one-sided adoption`

`+ standalone local verification`

The next major architecture direction is purpose-limited disclosure.

The long-term relation is:

`declare once`

`resolve only what the structure supports`

`disclose only what the purpose requires`

`preserve supported change without erasing origin`

`verify through reconstruction`
